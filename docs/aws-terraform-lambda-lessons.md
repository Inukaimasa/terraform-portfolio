# AWS / Terraform / Lambda で今回覚えておくこと

## 1. 成功 = 200 とは限らない

覚え方はこれです。

「その処理の目的どおりなら成功」

たとえば:

- `200` = 正常に結果を返した
- `201` = 正常に作成した
- `204` = 正常だが返す本文がない
- `302` = 正常に別URLへ飛ばす
- `400` = 入力が悪い
- `403` = 権限はあるが許可しない / 無効
- `404` = データがない
- `500` = サーバ側の想定外エラー

今回の redirect Lambda は、URLへ飛ばすのが目的なので `302` が成功。

---

## 2. 「設定」と「権限」は別

AWS ではかなり重要。

今回でいうと:

- `LINK_TABLE_NAME`
  → どのテーブルを見るかという設定
- `dynamodb:GetItem`
  → そのテーブルを読んでよいという権限

つまり、

- 名前を知っているだけでは触れない
- 環境変数があるだけでは動かない

この考え方は全部に効く。

例:

- S3 バケット名を知っていても `GetObject` 権限がなければ読めない
- Lambda 名を知っていても `InvokeFunction` 権限がなければ呼べない
- Secrets Manager のシークレット名を知っていても `GetSecretValue` 権限がなければ取れない

---

## 3. イベントの形が合っているかが大事

Lambda は「コードが正しいか」だけでなく、
入力イベントの形が合っているかで動作が変わる。

今回もそうだった。

### ダメだった例

    {
      "short_code": "test123"
    }

### 合っていた例

    {
      "queryStringParameters": {
        "shortCode": "test123"
      }
    }

つまり、

値があるかではなく、
コードがどの場所から値を読むかが大事。

これは API Gateway, EventBridge, SQS, SNS, Step Functions でも同じ。

---

## 4. ログを見るときは「見た目のメッセージ」より「本当の原因」を見る

今回も最初はこれだった。

    {
      "message": "Failed to read link data"
    }

でも本当の原因はログの下にあるこれだった。

    AccessDeniedException

つまり、

- 画面に出るメッセージ = 表向き
- スタックトレース / 例外名 = 本当の原因

### 覚え方

1. まず最後の error 文を読む
2. その次に exception 名を見る

今回なら `AccessDeniedException` を見た時点で、
コードではなく IAM を疑うべきだった。

---

## 5. 404 は悪ではない

404 は「壊れた」ではなく、
その入力に対応するデータが無かっただけ。

今回の流れ:

- `shortCode` なし → `400`
- `shortCode` あり、でも DynamoDB にデータなし → `404`
- データあり → `302`

この違いが見えるようになると、かなり実務っぽい。

---

## 6. Terraform は「一発で完成」ではなく「差分を見ながら育てる」もの

Terraform はだいたいこれの繰り返し。

1. 書く
2. `fmt`
3. `validate`
4. `plan`
5. `apply`
6. 動作確認
7. 足りない権限や変数を足す

最初から100点を狙うより、
最小で通してから不足を足す方が正しい。

今回も実際にこうだった。

1. Lambda 作成まで通す
2. 環境変数不足が出る
3. IAM 権限不足が出る
4. データ不足が出る
5. 最後に 302 成功

これはかなり自然な進み方。

---

## 7. module は「便利な魔法」ではなく「部品化」

今回の理解で大事なのはここ。

- `root` = 注文する
- `child module` = 作る
- `module output` = 返す
- `root output` = 最終表示する

この考え方は Lambda 以外でもそのまま使える。

たとえば今後:

- `modules/s3_bucket`
- `modules/apigateway`
- `modules/ecs_service`

みたいに広げても同じ。

---

## 8. 名前は揃える

地味だがかなり重要。

今回も:

- `lambda_name`
- `function_name`
- `environment_variables`
- `ebvironment_variables`

みたいな差で詰まった。

Terraform と AWS は、
意味が同じでも文字が違えば別物。

### 実務でのコツ

命名規則を早めに決めること。

例:

- Terraform 変数は `lambda_name`
- module output は `lambda_name`, `lambda_arn`
- 環境変数は大文字 `LINK_TABLE_NAME`

こういうルールを決めるとかなり減る。

---

## 9. 動作確認は「作成確認」と「実行確認」で分ける

### 作成確認

- `terraform apply` が通る
- Lambda が存在する
- Log Group ができる

### 実行確認

- 実際にイベントを投げる
- ログが出る
- 期待する status code が返る

作成できただけでは、まだ半分。
今回みたいに 作成成功 → 実行時に AccessDenied は普通にある。

---

## 10. 「どこで落ちたか」を段階で見る

今回の流れを分解するとこうだった。

1. Lambda 自体がない
2. Lambda はあるが環境変数不足
3. 環境変数はあるが IAM 不足
4. IAM はあるがデータなし
5. データもあり 302 成功

この見方ができると焦らなくなる。

エラーが出たら毎回これを考える。

- そもそもリソースはあるか
- 入力は合っているか
- 設定はあるか
- 権限はあるか
- データはあるか
- コードロジックは正しいか

かなり切り分けしやすい。

設定があっても権限がなければ動かない

## 3. 入力イベントの形がズレるとコードは動かない

値の有無より、どこから読むかが大事
